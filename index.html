Git Module Manager

Simpler, faster, less complex dependency management!

View on Github

Motivation

At some point, a project becomes complex enough that it can be broken down into sub-projects. Many package managers try to address this problem, but at the cost of added complexity and reduced performance. Git was designed to solve this problem, but it does so at a very low level.

Git's submodules receive much praise and criticism. Yet both depend on the interpretation of how they should be used; gmm focuses entirely on providing a workflow for dependency management.

Editing submodules is not an easy git workflow to manage. It's also the basis of most submodule criticism. Instead, they can be used as immutable dependencies. When you need changes, publish them from upstream.

Performance and availablility are important as important as usability! Every time you download a module, it should go to your cache. This makes offline time easier and installing modules faster since with git only the delta is downloaded.

Download

Written in bash so that it can run anywhere. Under 500 lines of code. Fewer moving parts means that fewer things will go wrong — easy to fork, grok and fix if something does.

Install using curl.


  (curl -sL https://goo.gl/kS3VRE > /usr/local/sbin/gmm && sudo chmod 700 gmm)

Install using git.


  git clone git@github.com:0x00A/gmm.git /usr/local/lib/gmm
  sudo chmod 700 /usr/local/lib/gmm/gmm
  ln -s /usr/local/lib/gmm/gmm /usr/local/bin/gmm

Install using bpkg.


  bpkg install -g 0x00A/gmm

Getting Started

Installing Packages


  gmm install name/repo

The install command will try to clone the dependency to a cache. It will then add the dependency from the cache to your project as a submodule.

You can specify a branch (branches can be named using semantic versioning). 


  gmm install name/repo 3.0.0

Installing usually means running a script from the package. For example, node.js projects have a package.json file with shell scripts in it, many C++ projects have a makefile, etc. gmm will try to do this for you.

Keep in mind, this is a generalized solution, not a drop in replacement for your package manager. The goal is to stay minimal and free of creep!

Settings

Configuring your workflow

Settings are managed though environment variables, so they can be saved without any special configuration files/formats and so that they can be set dynamically. You can find a full list of settings here. 


  exports HOST="mygitserver.com"

  HOST=github.com gmm install name/repo
